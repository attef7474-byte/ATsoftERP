Documented Limitation
---------------------
1. No backend unified search endpoint exists. Strategy A (frontend adapter-only)
   approach uses individual entity list/search endpoints. This means the modal
   makes up to 25 API calls per search (in batches of 5). Acceptable for admin
   intranet usage but not for high-traffic scenarios. A future backend endpoint
   could consolidate this.

2. Detail routes for some entities (e.g., barcodeLabel, auditLog, notification)
   may not have dedicated detail pages yet. The route is defined but the page
   may show a 404 until the page is created.

3. Screenshots were not captured due to environment constraints. Visual
   verification was done by confirming the build compiles and the component
   renders without errors.

4. Permission-aware filtering (hiding entity types based on user permissions)
   is not implemented at the frontend level. The backend API enforces
   permissions — if a user lacks read permission for an entity, the API call
   will fail silently and that entity type won't appear in results. This is
   intentional to avoid duplicating permission logic.
