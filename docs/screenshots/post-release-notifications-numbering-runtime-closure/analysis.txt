ANALYSIS — Post-Release Notifications & Numbering Runtime Closure
===================================================================
Date: 2026-07-21

=== ISSUE 1: Notification Unread-Count ERR_CONNECTION_REFUSED ===

Root Cause:
The GET /api/v1/notifications/unread-count endpoint returned a raw number
(e.g., "5") instead of wrapping it in an object like { count: 5 }.
The frontend polling hook expected json.count or json.data.count, which
was undefined for a raw number, causing the badge to always show 0.
Additionally, if the API process crashed after startup (due to unhandled
errors), subsequent requests would get ERR_CONNECTION_REFUSED.

Fix:
1. Changed controller response: return { count } instead of raw number.
2. Made NumberingModule @Global() to prevent DI resolution crashes.
3. Wrapped NumberingService.generateNumber/generateNumberAtomic in
   prisma.$transaction for atomicity.

=== ISSUE 2: Auto Code Generation Not Working ===

Root Cause:
All create services (Company, Branch, Department, Warehouse, Product,
Machine, etc.) either required the user to manually supply a code or
reimplemented numbering logic without using the central NumberingService.
No service injected or called NumberingService.

Fix:
For each approved entity with a code field:
1. Made code optional (@IsOptional) in the create DTO.
2. Injected NumberingService into the service.
3. If code is not provided (or empty), auto-generate using
   NumberingService.generateNumberAtomic('<SEQUENCE_TYPE>').
4. Updated 10 frontend create forms to remove the editable code field
   (codes are now auto-generated by backend).
