1. REASON GLOBAL VALIDATIONPIPE WEAKENING WAS REJECTED
   - The previous audit set whitelist: false, forbidNonWhitelisted: false globally to allow free-form JSON bodies on settings PATCH endpoints.
   - This weakens API security by allowing arbitrary unknown fields into every DTO across the entire application.
   - The NestJS ValidationPipe whitelist feature is a critical security boundary that strips unknown properties.
   - Disabling it globally opens the door to mass-assignment, extra-field injection, and schema-poisoning attacks.
   - Permissions controllers, user creation, audit logs, and all other endpoints also lost whitelist protection.
   - The correct approach is per-DTO explicit field definitions, not global pipe weakening.

2. CONFIRMATION MAIN.TS RESTORED TO SECURE VALIDATION
   - apps/api/src/main.ts: global ValidationPipe restored to:
     whitelist: true
     forbidNonWhitelisted: true
